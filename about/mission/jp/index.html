WaSP: スタンダードに向かって

The World Wide Web Consortium (W3C)は、他のグループやスタンダード団体とともに、ウェブベースのコンテンツを作成、また理解するための技術を作り上げてきました。「ウェブスタンダード」と呼んでいるこれらの技術は、ウェブ上の全ての文書の長期にわたっての生存能力を維持するかたわら、より多くのウェブユーザーにより多くの利点を与えられるよう、細心の注意を払ってデザインされています。詳しくはサイドバーをご覧下さい。

このようなスタンダードをおいてデザインや制作をすることにより、より多くの人々とより多様なインターネットデバイスに対応するサイトができ、その上コストは削減され、制作自体は単純化されます。こういった形で作られたウェブサイトは、これまでのデスクトップブラウザの進化、また新たなインターネットデバイスの登場にも関わらず、変わらずに機能するものとなります。

非常にわかりやすく良い考えですよね。では何が問題なのでしょうか？また、なぜウェブスタンダード・プロジェクトは存在するのでしょう？

問題点

W3C ができて以来ブラウザ企業大手はウェブスタンダードの設立に関わってきましたが、何年もの間準拠は見られませんでした。スタンダードの統一サポートが欠落したブラウザを公開することによって、ブラウザメーカーは不必要にウェブを断片化し、デザイナー、デベロッパー、ユーザー、そしてビジネスともを傷つける結果となりました。

消費者は W3C スタンダードの統一サポートの欠落に不満を抱きました。「間違った」ブラウザを使用した際、コンテンツが見られなかったり、希望する行動が行えなかったりする人が多数出ました。このうち特に被害をうけたのは、障害や特別な必要性のある人々でした。

苦境とコスト

それと同時に、W3C スタンダードの統一サポートの欠落はデザイナーやデベロッパー、サイトの持ち主を大変な苦境に陥れました。 スタンダードに未対応のブラウザのために、全てのページを数バージョン作る予算があるだろうか？ないとしたら、どのブラウザを無視し、それにより何百万のユーザーをかえりみない覚悟があるのか？どちらをとっても、そのコストは莫大なものでありました。これはいまだ変わっていません。

破砕されたブラウザ市場は、全サイト制作のコストを少なくとも 25 ％ 増加させました。予算の不足により、多くのデベロッパーは顧客となる可能性のある人々を閉め出すサイトを作ることとなりました。スタンダードを知るデベロッパーの多くは、それに未対応のブラウザのためにサイトを作ることに意義を見いだせなかったのです。それ以外の人々は、スタンダードのことをよく、または全く知らずにいました.こういった人々は今も変わらず多く存在しますが、その中には ASP、Java、Flash MX、そして .Net までもを把握しているかに見られるにも関わらず、構造的マークアップやスタイルシート、そして構造と体裁とを区別することの重要性を全くといって理解していない何百ドル規模の大企業も含まれます。

デザイナーの一部には、ブラウザの未対応性に嫌気がさし、最古で最もサポートされているウェブ技術以外は自らのサイトから故意に抹消した者もいます。そのようなサイトは、全てのブラウザで機能することには成功しましたが、これは限られた消費者にとってのみのアピールと機能性との引き替えに手に入れたものでした。

他のデザイナーは、ビジュアルエディターや制作ツールに依存し、人気のある様々なブラウザの癖に合わせて最適化された、複数のマークアップやコードを生産しました。これは帯域バンド幅（帯域幅）だけでなく金銭的にも無駄を出した上、次世代のブラウザでは動かなくなることも（スクリーン・リーダーからリナックス、PDA や使用者の少ないオベラなどのブラウザといったオルタナティブ・ブラウザやデバイスでは、全く初めから動かなかったりも）しました。ウェブは、一度は印象的だったが近代的なブラウザやデバイスでは機能不可能なサイトの死骸で散らかっています。さらに悪いことに、そういったサイトは今でも毎日生み出されているのです。

デザイナーの中には、あまりに不満がつのったためにウェブスタンダードにまるきり背を向け、専売環境専門に制作し始めた人もいます。クリエイティビティ面の可能性は豊かであるにも関わらず、そういった技術は広範囲にわたるアクセシビリティの欠如の被害を受け、ブックマーク（お気に入り）や印刷、コピー＆ペーストその他、情報サイトや業務処理サイトに不可欠な一般的要求を満たすことができなくなります。

必要性からの誕生

このような問題に応え、ウェブスタンダード・プロジェクト (WaSP) は 1998 年、ウェブスタンダードの核心を広め、ブラウザメーカーにもその促進を奨励することにより、シンプルで手頃なアクセスを皆に確保することを目標に編成されました。

私たちのメッセージは、当初（特にブラウザ会社のマーケティング部と広報部から）抵抗を受けましたが、多くのブラウザ企業のエンジニアが私たちに協賛し、WaSP を経営者との内部抗争においての味方として見てくれたこともあり、最終的には説得することができました。

2000 年を初めに、私たちが（時にはしつこいほど）推奨してきたスタンダードの多くを約束するトップのブラウザが次々と登場しました。現在の市場トップのブラウザは、そのライバルとともに HTML 4、XHTML 1、CSS、ECMA スクリプト (Java スクリプトの標準バージョン)、そして DOM の素晴らしいサポートを備えている、または近く備えるところまできています。

こういったブラウザのおかげで、ようやくデザイナー、デベロッパーは XHTML と CSS を用いて制作にかかることができ、多くの場合、構造と体裁を分離しポータビリティとアクセシビリティを最大限にすることができるようになりました。気を配れば、 デザイナーとデベロッパーはW3C スタンダードの DOM を使って洗練された動きをサイトに加えることもできるのです。

ではいったい何が問題なのでしょうか？なぜウェブスタンダード・プロジェクトはいまだ存在するのでしょう？

これからの難問

今日のブラウザはスタンダードをサポートしますが、何千、何万ものプロのデザイナーやデベロッパーは構造と体裁をいっしょくたにし、時には意味的構造を避けて、 (X)HTML をデザインのツールとして使ったりといった、時代錯誤の方法を使い続けています。高給をとるプロが、構造的に無意味なマークアップ、巨大イメージマップ、重ねに重ねられたテーブル、あまりに古いため、もともと解決するのが目的だったユーザビリティの問題をかえって引き起こしているスクリプトなどで構成された、スタンダードに不適応でアクセスしづらいサイトを作り続けているのです。

ウェブ制作についての本の多くはいまだに旧式の方法について書かれており、多くの人々は、スタンダードをサポートするブラウザとしないブラウザのどちらを使っても、まるきり同じように見えて動くサイトにプライドを持っているようです。それによりアクセシビリティや長期の生存性、将来へのサポートが犠牲になるというのもかえりみずです。片手で数えられるほどの人気ブラウザでしか動かない専売のコードを書く人もいます。

ですから WaSP の主要目標は、私たちの仲間が自分たちのため、顧客のため、そしてサイトのユーザーのためになるスタンダード準拠の方法を学ぶための教育資源となることです。

たくさんのプロがブラウザ戦争のピークのビジュアルエディター環境のなかで仕事をこなしてきました。上で書いたように、こういったツールはもともとスタンダードのかわりに 4.0 ブラウザの特異な癖に最適化した、スタンダードに不適合で意味的でないサイトを作るようにできているのです。2002 年、二つのトップ・ビジュアルエディターがウェブスタンダードとアクセシビリティのサポートを大幅に改良しました。（そのうちの一つはウェブスタンダード・プロジェクトが改良に協力しました。）しかし、この改良を生かすためには、ウェブスタンダードを使ってデザイン、制作することの基本と利点をプロが学ばねばなりません。この点もまた、デベロッパーの教育の必要性を示唆します。

今日のブラウザやデバイスでアクセスでき、これからそれらが進化しても生き残れるサイトを求めるなら、顧客やサイトの管理者にもこの情報が必要になります。スタンダードの利点を一度知ったら、サイト管理者は自分らのサイトをどんな環境でもまるで同じである印刷広告として見るのをやめてくれることが WaSP の望みです。そしてそのかわり、個々のブラウザやデバイスのニーズと性能に合わせて、見た目は多少変わろうとも適応したコンテンツと機能性を提供することに重点をおいてくれれば、と思います。

どの「スタンダード」？

「スタンダード」とは、次のことを意味します：

構造化言語
拡張可能ハイパーテキスト・マークアップ言語：Hypertext Markup Language (HTML) 4.01
拡張可能ハイパーテキスト・マークアップ言語：Extensible Hypertext Markup Language (XHTML) 1.0
拡張可能ハイパーテキスト・マークアップ言語：Extensible Markup Language (XML) 1.0
プレゼンテーション言語
カスケーディング・スタイルシート：Cascading Style Sheets (CSS) レベル 1
CSS レベル 2 revision 1
CSS レベル 3
オブジェクトモデル
ドキュメント・オブジェクトモデル：Document Object Model (DOM) レベル 1 （コア）
DOM レベル 2
DOM レベル 3 (コア)
スクリプト言語
ECMA スクリプト 262 (Java スクリプトの標準バージョン)
