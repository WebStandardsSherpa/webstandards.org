1 序論

Tim Berners-Lee の、自らの発明 World Wide Web に関する夢は、それがユーザーが共に活動し、遊び、仲良くするための情報を分かち合う場所となること。(World Wide Web 超短個人的歴史)ウェブデベロッパーとして、ビジネス関連、社会関連、また教育関連のサイトをつくることで、私たちはこの夢を実現します。

しかしこの急速な成長期、ウェブにはその可能性を実現するための指導が必要です。ウェブスタンダードこそがこの指導なのです。これらのスタンダードは皆が私たちの提供する情報にアクセスできることを保証し、またウェブ制作を速く、より楽しめるものにもします。

スタンダード準拠は、特別な必要性のある人々がウェブを使うのを楽にするのに役立ちます。目の見えない人は、コンピューターにウェブページを読ませることができます。視力の弱い人は、読みよくするため、ページレイアウトを並べ替えたり、ズームすることができます。そしてハンドヘルド・デバイスを使う人は高性能の仕事環境を持っている人と同じように楽にウェブを見ることができます。

これから説明するように、デベロッパーがウェブスタンダードに気を配るべき実用的な理由もたくさんあります。例えば、サーチエンジンはサイトをより効率的に巡回できます。 各ブラウザ特有のコードは、ウェブページを作る際の手間を二倍にも三倍にも増やし、 新たな媒体が出てきた時には非常に不満が残る結果となります。こういった状況は、ウェブスタンダードの的確な指揮なしでは悪化するばかりです。

スタンダードは制約的なのではないかと懸念する人がいます。実際には、スタンダードによってウェブ制作に関わる面倒な作業の多くは除去され、デベロッパーはより自由にクリエイティブな面に時間をかけることができます。スタンダードは将来の進歩と過去の技術への留意の双方に対応しているのです。

今日夢としか思えないものも含め、ウェブの多くの使用法は、スタンダード準拠の広まりなしには不可能、またはより困難なものになるでしょう。現時点では、普遍的に近いとさえ思われるほど非常に一般化しているシステムやソフトウェアが存在しますが、明日がどうなるかなど誰にわかるでしょう？どこか一つの企業のコントロールに自らを拘束するということは、その一社の供給しえる、あるいは供給するものの幸運と不幸とに私たちの将来を制約することを意味します。普遍的スタンダードの維持により、ウェブが今日のペースで革新を促しつつ生き残ることが可能になるのです。

スタンダードの提供できるものはあまりにも多いため、ウェブスタンダード・プロジェクト (WaSP) の私たちはあなたがスタンダードについて学ぶことを手助けせずにはいられません。この文書は単なる出発点でしかありませんが、どんなスタンダードが存在するか、なぜ存在するのか、そしてなぜそれが大事なのかを確実に理解することができるでしょう。ウェブの一部分を作るたびに、私たちは共有の情報スペースであるウェブというものに貢献しているのです。育てていくこともできれば、重荷を加え、その破壊を助長することもできます。その選択は私たちに、そしてその結果は全ての人に帰属するのです。


2 スタンダードとは


2.1 W3C スタンダードとは


2.1.1 W3Cとは

World Wide Web Consortium (W3C) は、ウェブの発明者である Tim Berners-Lee の率いる「ウェブを最大限に生かすこと」に専念している国際的産業コンソーシアムです。1994 年に設立され、マイクロソフト、アメリカ・オンライン （ネットスケープ・コミュニケーションズの親会社）、アップル・コンピューター、アドビ、マクロメディア、サン・マイクロシステム、その他様々なハードウェアやソフトウェアの製造メーカー、コンテンツ・プロバイダー、教育機関、通信企業を含めた、450 以上ものメンバー団体を有します。このコンソーシアムは、アメリカの MIT、ヨーロッパの INRIA 、日本の慶応大学の三つの研究機関に主催されています。


2.1.2 W3C の役割

The W3C は、ウェブ関連製品の相互運用性を向上させるための公開仕様書（事実上のスタンダード）を作成します。 W3C の勧告はコンソーシアムのメンバーと、要請を受けた専門家から成り立つ運営グループによって作られています。運営グループは、ウェブアプリケーションのメーカーである企業や他の組織から一般的合意を受け、草案と勧告案を作成します。これらは W3C 勧告として W3C メンバーと管理者の公式承認を受けます。この過程と検閲段階についての詳細は W3C のサイトでご覧いただけます。


2.1.3 W3C スタンダードって何?


2.1.3.1 HTML 4.0 – ハイパーテキスト・マークアップ言語：HyperText Markup Language

ハイパーテキスト・マークアップ言語：HyperText Markup Language (HTML) はウェブ上でテキスト文書に構造を加えるために広く使用されています。ブラウザはこの文書を解釈し、構造をユーザーの媒体にあわせて表現します。例えば、視聴覚的ブラウザが通常 strong という要素 (<strong> … </strong>) を太字で表すのに対して、音声ブラウザは読むときにその部分を強調するといった具合です。

カスケーディング・スタイルシート (CSS) のおかげで、デベロッパーは構造要素がどのように表現されるか、ブラウザの初期設定を書き換えて指定することができます。


2.1.3.2 XML 1.0 – 拡張可能ハイパーテキストマークアップ言語：Extensible Markup Language
XML ドキュメントの一部の例 
<addressbook>
  <entry>
    <name>Bill Gates</name>

    <email>bgates@microsoft.com</email>
  </entry>
  <entry>
    <name>Marc Andreesen</name>

    <email>marca@netscape.com</email>

  </entry>
  <entry>

  </entry>

</addressbook>

拡張可能ハイパーテキストマークアップ言語：Extensible Markup Language (XML) は HTML
のようなマークアップ言語ですが、一定の決まった要素が用意されているのではなく、自分で要素の指定、または誰かが作成したものの利用ができるようになっています。XML 名前空間 を使うことにより、ひとつのドキュメントの中で、複数のセットを使用することさえ可能です。

XHTML や MathML のように、 XML の適用例のいくつかはすでに W3C の勧告となっています。他のものは現在 W3C 草案です。

CSS や XSL のようなスタイルシート・スタンダードは、XML 要素がどのように表されるかを指定する様々な選択肢を提供します。ブラウザの XML の直接の表現に対するスタンダード準拠サポートは一定していないため、人間に情報を提供するには、 CSSを使ってスタイル指定をした HTML (または XHTML) が最適でしょう。今日 XML は主にマシーン間での通信に使われています。

XML は、主に自分で要素を付け加えたり構造システムを作ったりすることが可能なため、 HTML よりも柔軟です。このため XML は大量のデータの整理に最適なフォーマットであり、すでに多くのデータベースやサーチエンジンに使用されています。


2.1.3.3 XHTML 1.0、1.1、モジュラリゼーション

XHTML 1.0 は XML の応用としての HTML 組成変更です。XHTML 1.0 は観念学的には HTML 4.01 からきていると見てよく、 XML の影響によって技術的にはより厳密だと言えます。

XHTML は相応の HTML と同様にブラウザに表示されます。コンテンツを再処理しなければならない、例えば PDA 用に再加工するといった可能性がある場合、 XML のより厳密な構文規則は、XHTML の自動処理を通常の HTML よりも楽に、安価にすることができます。

観念学的には、 XHTML 1.0 は以下の HTML 4.01 一般概念を受け継ぎます：

  • 体裁と文書構成がスタイルシートによって分離されている
  • 文書がアクセシブルに作られている
  • 文書が国際化されている

またXHTML 1.0 は Strict、Transitional、Frameset の三つの DTD モデルを用います。このモデルは HTML 4.0 で生まれ、その後 HTML 4.01 にも使用されています。

XML から XHTML への重要な技術的慣例は以下を含みます：

  • 全ての文書タイプが的確な DOCTYPE（文書）宣言によって宣言されている
  • 合致する文書の構造が DOCTYPE 宣言、XHTML 名前空間を宣言した html 要素、title を含んだ head 要素、body 要素を含む
  • 要素と属性が全て小文字であり、全ての属性値が引用符で囲んである
  • 全ての非空要素（例： p、li）が終了タグできちんと閉じられている
  • 全ての空要素（例： br、hr、img）がスラッシュ (/) できちんと閉じられている (<br />)
  • 文書が宣言された DTD 検証で承認される

テンプレートは 学ぶ > テンプレートでどうぞ。

XHTML 1.1 は三つの主要部分から成っています：

  • XHTML 1.0 Strict DTD (多少の変更あり)
  • XHTML モジュラリゼーション
  • ルビ注釈

XHTML 1.1 のドキュメントを作成したいなら、いくつかの方法があります。一つめは public XHTML 1.1 DTD を使う方法。これは、XHTML 1.1 の体裁属性がほぼ全く無いため非常に体系的にまとめられます。 構造と体裁の分離は完結され、体裁に関する情報は全てスタイルシートで扱われます。

XHTML 1.1 で文書を作成するもう一つの方法は、XHTML モジュラリゼーション利用です。これは慣れ親しんだ HTML の構成要素（テキスト、テーブル、フレーム、フォームといったもの）と XHTML を個々の塊に分割することです。これにより、自分の DTD を書き、必要な要素のみ使用することができるようになります。これがまさに拡張の実行であり、ウェブ作者であるあなたは、実質的に自分のマークアップをカスタマイズする機会を持つことになります。

ルビ注釈は一部のアジア文字に使われる注釈の振りかたに対応する特殊な手段です。これは国際化の作業と見なされます。


2.1.3.4 CSS – カスケーディング・スタイルシート

カスケーディング・スタイルシート：Cascading Style Sheets (CSS) は、要素のタイプ、自分で指定した要素のクラス、または個々の事例に対してスタイルをあてることによって HTML または XML 要素の外見を変更するメカニズムです。

スタイルシートはサイト全体の外見を一貫して定めるのに役立ちます。W3C は、CSS の誕生に続き、レイアウト限定の HTML 機能を除々に廃止し、スタイルシートに置き換えることによって、よりシンプルで体系的なウェブを目指すことを推奨しました。


2.1.3.5 DOM 1 – ドキュメント・オブジェクト・モデル：Document Object Model レベル 1

DOM は、スクリプト言語（Java スクリプトの標準バージョンである ECMA スクリプトのようなスクリプト言語）の勢力とインタラクティビティをウェブページ上で最大限に生かします。（プログラミング用語では、ドキュメント・オブジェクト・モデル：Document Object Model (DOM) レベル 1 は、ウェブページとやりとりするアプリケーション・プログラミング・インターフェイス (API) です。）DOM により、HTML や CSS などで書かれた文書の構成、コンテンツ、体裁にスクリプト言語でアクセスすることが楽になります。

DOM は技術の将来的進歩に適応しています。どんな言語が使われている文書であっても、全てのスクリプト言語とその文書とのインタラクティビティを可能にするでしょう。このスタンダードはダイナミック HTML をプログラミングすることを楽にするだけでなく、将来のインターネット技術への適応に対する苦痛も和らげることでしょう。


2.2 ECMA スタンダード


2.2.1 ECMAとは

欧州電子計算機工業界：European Computer Manufacturers Association (ECMA) は、プログラミング言語や出入力コードも含めたコンピューターの演算形式の標準化への需要を満たすため、1961 年に正式に設立されました。

ECMA はスイスのジュネーブ、 国際標準化機構：International Organization for Standardization (ISO) と国際電気技術委員会：International Electrotechnical Commission (IEC) の本部近くに拠点を置いています。1994 年、そのより広範囲な活動内容を反映するよう ECMA・European Association for Standardizing Information and Communication Systems に改称されました。


2.2.2 何をするの？

ECMA の主な役目は情報技術や通信技術といった分野に関するスタンダードと技術報告書を開発することです。公式な標準化機関ではなく企業の集まりであるため、ECMA は国内外の公式機関と協調して取り組むことが頻繁にあります。

ECMA スタンダードは、国際標準と欧州標準の基盤として受け入れられてきました。これまでに、270 以上の ECMA スタンダードと 70 以上の技術報告書が発表されています。

このうち 85 は国際標準化機構：International Organization for Standardization (ISO) から国際標準として認定されています。さらに、25 は欧州通信規格協会：European Telecommunications Standards Institute (ETSI) に欧州標準として認められました。


2.2.3 ECMA スタンダードとは


2.2.3.1 ECMA スクリプト (Java スクリプトの標準バージョン)

ECMA スクリプトはネットスケープの Java スクリプト と マイクロソフトの J スクリプトを主として基盤とした標準スクリプト言語です。ECMA スクリプト・スタンダードは ECMA の技術委員会 39 (TC-39) によって定められています。

オブジェクトベース言語である ECMA スクリプトは、主にドキュメント・オブジェクト・モデル (DOM) によって指定されたウェブページ上のオブジェクトを処理するのに使用されます。このオブジェクトは、（ウェブページの構成要素またはページ全体のことになりますが、）追加、削除、移動させることができ、属性の変更などが可能になります。これにより、ウェブデベロッパーはテキストをアニメーションにしたり、グラフィックにロールオーバー効果をかけたり、ユーザーの入力に基づいて再ロードなしで変化するページを組み込んだりといったことができるようになります。

現在の ECMA スクリプト仕様書は、標準 ECMA-262、ECMA スクリプト言語仕様書第二版です。


3 ウェブスタンダードの利点とは


3.1 アクセシビリティ


3.1.1 ソフト、マシンにとって

ウェブスタンダードに準拠することで、ウェブページはウェブサーチでより良い位置が占められるようになります。準拠した文書の構造情報はサーチエンジンでアクセスしやすく、情報の評価が楽なため、こういった文書の情報はより正確に索引をつけることができます。

ウェブスタンダードの使用は、クライアントサイドに加えてサーバーサイドのソフトウェアによる文書の構造の解釈も楽にし、サイトへのサーチエンジンの追加を楽に、またより良い結果がでるものにします。

スタンダードは古いブラウザでも文書の基本的構造を理解できるように書かれています。新しく追加された「クール」なスタンダードを理解することができなくても、コンテンツは表示することができるのです。もちろん、同様のことがサーチロボットにも言えます。サーチロボットとは、サーチエンジンや他の検索プログラム用に、サイトに掲載された情報を収集するシステムであるロボットのことです。

スタンダード準拠のコードは、自分のページを検証サービスで確認する機会を与えてくれます。検証サービスは文書をプロセスし、エラーリストを提示します。これによりエラーを見つけ訂正するのが随分と楽になり、また時間の節約にもなります。

スタンダードに準拠した文書は、簡単に他のフォーマット、例えばデータベースやワード文書に変換できます。これはウェブ上の文書中の情報の多用途の使用を可能にし、テレビや PDA のようなデバイスを含むソフトならびにハードウェアといったシステムの、新しいものへの移行を単純化します。


3.1.2 人にとって

アクセシビリティー は多くのウェブスタンダード、特に HTML を支える重要な発想です。

障害のある人々がウェブを使えるようにするというだけでなく、視力障害のある人にウェブページを音読する音声ブラウザ、テキストを点字に変換する点字ブラウザ、非常に小さな画面のハンドへルド・ブラウザ、文字多重放送その他特殊な出力デバイスを含む、なじみのないブラウザを使用する人々にも使えるサイトにすることができます。

ウェブへのアクセス手段が増加するにつれ、全てのニーズに応えられるようにするためサイトを調整したり、何種かバージョンを用意したりといったことはますます難しくなるでしょう。（実際、今日でさえ無理であると言う者もいます。）スタンダードを追うことは、この問題解決への大きな一歩なのです。サイトをスタンダード準拠にすることは、新旧あわせた従来のブラウザが現存のサイトを正確に表示するのを保証するだけでなく、変わったブラウザや媒体でも動くことをも約束します。

スタンダードを無視した結果のいくつかは明らかなものです。最も基礎的な結果は、サイトへのアクセスが制限されることです。サイトを使いたいと希望するユーザーのうち、ほんの一握りのみにアクセスを限定するというのは、ビジネス面から言ってどれだけ理にかなったものでしょう？ビジネスサイトにとって、たとえ少数であってもターゲットユーザーを拒絶することは、利益に大きく影響します。教育サイトにとっては、裕福で障害がなくグラフィック・ブラウザを持った生徒だけでなく、第三世界諸国でテキストベースのブラウザにしかアクセスのない子供や、特別なブラウザを使っている障害のある学生にも利用できるものにすることは筋が通っています。

同じ原理が全てのタイプのウェブサイトに適用できます。スタンダードから離れ、決まったブラウザ特有の機能を活用するのは魅力的かもしれませんが、スタンダード準拠からくるアクセシビリティの向上は、長期的に見るとはるかに優れた見返りにつながります。


3.2 安定性

ほとんどのウェブスタンダードは通常、旧バージョンのスタンダードを使ったデータも新ブラウザで機能し続け、また新バージョンのスタンダードを使ったデータは旧ブラウザでも許容範囲の結果を出し、「上品に格が下がる」ようにするために、下位互換性と上位互換性を念頭に置いてデザインされています。

ウェブサイトはその寿命のうちにいくつかのデザイナーチームに扱われる可能性があるため、コードが理解しやすく、簡単に更新できることが重要です。ウェブスタンダードは、どんなウェブデベロッパーでも従い、理解し、慣れることのできるルールを提供します。あるデベロッパーがスタンダードに沿ってデザインしたサイトを、その途中から別のデベロッパーが引き継ぐことができるのです。


4. 結論

ウェブデベロッパーとして、私たちは違ったブラウザの違ったバージョンによるウェブページ表示の不統一性という問題に対処しようと常に努力しています。これにより、時間を消費する二種あるいは多種のコードを書くこと、または大衆の一部の利用を不可能とまではいかなくても困難なものにする、一種類のブラウザのみ対応のコードを書くことが必要になるのです。こういう事態は、電話、ポケベル、PDA といったその他のハードウェアとソフトウェアの出現によりさらに悪化するでしょう。

ウェブスタンダードは、非世俗的な組織に定められた不可解な法律ではありません。説明した通り、ブラウザ・メーカー、ウェブデベロッパー、コンテンツ・プロバイダーやその他、スタンダードを実際使用する人々の代表によってその大部分は決定されているのです。

スタンダードに沿ってウェブページを作成することは、サイトの制作時間を短縮し、メンテナンスを楽にします。デバッグやトラブルシュートするのがスタンダードに合ったコードであるため楽になります。同じ体裁を達成するように何バージョンものコードを書いたり、メンテナンスすることの心配はなくなります。ひとつのバージョン、それで充分なのです。

ウェブスタンダードが世界共通に受け入れられることは、傑出した重要性を持ちつつあります。ウェブスタンダード・プロジェクトの方針は、ブラウザ・メーカーやウェブエディタ・メーカーがアプリケーションをスタンダードに沿ったものにすることの推奨で、ウェブデベロッパーとユーザーにとってウェブをより良い場所にすることです。ウェブデベロッパーがスタンダードを当然として使用し、コードのジェネレーターやレンダラーがスタンダードに順応することを強く主張することは、この努力の支援となります。

私たちの挙げた理由は、ウェブデベロッパーであるあなたにスタンダードを取り入れる充分な動機を与え、仕事場やデベロッパー仲間にスタンダードを勧める充分な兵器となるでしょう。

夢の実現にご協力ください。
